# 里程碑 4 (M4) 详细开发计划：DM 后台、动态图谱与产品闭环

## 阶段目标
这是打造终局壁垒的阶段。实现动态世界的高级运转（如角色死亡导致图谱变动），并为开发者提供透视这一切的“DM 上帝之眼”。

## 预估周期
4 周

## 第一周：世界状态动态覆写 (Dynamic Lore Overwrite) 逻辑
* **目标：** 让“蝴蝶效应”成为可以被系统化记录和展示的底层状态。
* **具体任务：**
  * [ ] **触发机制设计**：当 Zep 的会话摘要达到一定的文字量/回合数时，主动将这段摘要抛给一个特殊的审核 LLM，判定是否发生了“剧情核心状态变更”。
  * [ ] **图谱节点操作实现**：如判定通过（例如某关键门派被铲除），则对当前 Session 独占的图谱进行覆盖写（增加/删除 Node/Edge）。这些设定将长期凌驾于本书的初始图谱之上。

## 第二周：叙事节奏控制器 (Pacing Controller) 构建
* **目标：** 解决玩家陷入“无聊聊天”或者“卡关”时的业务干预。
* **具体任务：**
  * [ ] **冗余闲聊检测模型**：后台记录玩家回合内的有效实体（Entity）改变率，若连续数轮回合无显著进展，阈值置红。
  * [ ] **突发危机池构建**：为当前小说设定一个简单的危机池或与收束模式结合在一起。当检测到拖沓时，主动将这些内容混入 Director Prompt 的“隐藏指令区”。

## 第三周：上帝视角 DM 查阅台面板 (Dashboard)
* **目标：** 让开发者或高端玩家透视并管控多维度的故事线索。
* **具体任务：**
  * [ ] **AI 思考透明度面板**：展示 Director 系统对当前局势的理解。如“当前所处派系阵营”、“隐藏剧情发展意图”、“预估下两步走向”的可视化输出。
  * [ ] **动态图谱 Diff 可视化**：将初始设定图谱与当前用户推演出来的异化图谱直接放在一起对比，红绿高亮显示哪些人和关系被玩家的行为实质性改变了。

## 第四周：压力测试、系统安全保障与完整发布
* **目标：** 清理死角，保障应用在高负载或不当提示注入下的稳定性。
* **具体任务：**
  * [ ] **并发回溯与流传输压测**：确保在多分支克隆（大量复制 Zep Context）时的接口和数据库承载力。
  * [ ] **安全防护/防越狱强化 (Prompt Shield)**：处理部分玩家可能会采取脱离设定的“越狱指令”，保障叙事框架不被恶意解构。
  * [ ] **产品上线发布**：整理所有产品体验的闭环细节，最终测试并部署到线上环境。

## 验收标准
1. 在复杂的游玩分支中，系统能够识别重大事件并精准改动图谱，并可直观地在开发查阅台中展示出设定图谱的变更对比。
2. 闲置或漫无目的时能正确触发节奏控制器；实现从技术框架验证到全面可游玩产品的升维闭环。
